# Architecture Comparison: `main` vs `ufen-mtfn-v2`

**Paper:** Cai et al., "Multimodal sentiment analysis based on multi-layer feature fusion and multi-task learning", *Scientific Reports* (2025) 15:2126.  
Both branches implement the UFEN + MTFN architecture on CMU-MOSI, but with substantially different design choices.

---

## Quick Summary

| Dimension | `main` branch | `ufen-mtfn-v2` branch |
|---|---|---|
| BERT role | Frozen feature extractor | Fine-tuned during training |
| Total parameters | ~442 K | ~110.8 M |
| Sequence handling | Fixed T=30, no padding masks | Variable-length, masked throughout |
| UFEN conv design | Sequential (stacked) branches | Parallel branches, different kernels |
| Self-attention gate | Custom manual MHA implementation | `nn.MultiheadAttention` |
| MTFN cross-attention | Shared per-modality weights (6 pairs share W_Q/K/V/O) | 6 independent modules |
| d_e (cross-att dim) | Separate `d_e=64` (decoupled from d_m) | Same as `d_m=128` |
| LR strategy | Single LR=5e-3 for all params | Dual LR: 5e-3 (non-BERT) + 2e-5 (BERT) |
| LR scheduler | None | Cosine annealing + BERT warmup |
| Visual feature dim | 35 (FACET 4.2) | 47 (different FACET version) |
| Best test MAE | 0.981 | **0.812** |
| Best test Corr | 0.618 | **0.745** |
| Best test Acc7 | 34.3 | **43.0** |
| Best test Acc2 (neg/pos) | 77.7 | **80.3** |

## 1. BERT Role

### `main` — Frozen feature extractor (pre-extracted)
BERT is **not part of the trainable model**. Text features are pre-extracted once using `bert-base-uncased` and stored as fixed (N, 30, 768) arrays in `.pkl` files. The model never touches BERT weights.

```
Source: docs/architecture.md §2:
"Note: BERT features are pre-extracted and frozen. The paper says 'BERT pre-trained model
as the feature extractor' without mentioning fine-tuning, so BERT is NOT part of the
trainable model."
```

### `ufen-mtfn-v2` — Fine-tuned end-to-end
`bert-base-uncased` is loaded in `MultiTaskModel.__init__` and called **live in every forward pass**. Its weights are updated during training using a separate low LR.

```python
# model.py
self.bert = BertModel.from_pretrained('bert-base-uncased')
# forward():
bert_out = self.bert(input_ids, attention_mask, token_type_ids).last_hidden_state
feat_t, y_t = self.ufen_t(bert_out, t_mask)
```

**Impact:** The fine-tuned branch has 110M trainable BERT parameters vs. ~442K in main. The continuous adaptation of BERT's contextual representations is a major reason for the performance gap.

**Paper's intent:** The paper's statement (Experiments section) "BERT pre-trained model as the feature extractor" is ambiguous. However, keeping BERT frozen leads to MAE=0.981, while fine-tuning reaches MAE=0.812 — strongly suggesting fine-tuning is what the paper authors intended.

## 2. Sequence Handling & Padding Masks

### `main` — Fixed T=30, no masks
All three modalities are pre-padded/truncated to a uniform T=30 at preprocessing time. No padding mask is ever passed to any module; the model treats all 30 time steps as valid.

```python
# src/dataset.py
self.text   = data["text"].astype(np.float32)    # (N, 30, 768)
self.audio  = data["audio"].astype(np.float32)   # (N, 30, 74)
self.visual = data["visual"].astype(np.float32)  # (N, 30, 35)
```

```python
# src/models/ufen.py — no mask argument
def forward(self, x: torch.Tensor):   # x: (B, T, D_i), no mask
```

### `ufen-mtfn-v2` — Variable-length, masked throughout
Each batch is padded to the longest utterance in that batch. Boolean padding masks are generated by `data_loader.py` and threaded through every component (BiGRU, Conv, Self-Att, Cross-Att, mean-pooling).

```python
# model.py — masked_mean for correct temporal aggregation
def masked_mean(x, mask=None):
    valid = (~mask).float().unsqueeze(-1)
    lengths = valid.sum(dim=1).clamp(min=1.0)
    return (x * valid).sum(dim=1) / lengths

# CrossModalAttention passes mask to MHA
out, _ = self.mha(x_i, x_j, x_j, key_padding_mask=mask_j)
```

**Impact:** `main` has spurious signal from zero-padding positions polluting mean-pools and attention scores, particularly for short utterances. The masked approach prevents this but adds complexity.

## 3. UFEN — Conv Branch Architecture

This is the most fundamental architectural difference. The two branches interpret the paper's "N parallel Conv1D branches" (UFEN, §3.2 of the paper) in opposite ways.

### `main` — Sequential (stacked) branches

Conv layers are **stacked**. The output of Conv_l feeds as input to Conv_{l+1}:

```
BiGRU output → Conv_1 → ReLU → Self-Att gate → Unpool → +sum
                  ↓ (raw conv output, before attention)
               Conv_2 → ReLU → Self-Att gate → Unpool → +sum
                                                            ↓
                                                        X_fusion
```

```python
# src/models/ufen.py
conv_input = gru_out.transpose(1, 2)   # starts as BiGRU output
for conv, self_att, unpool in zip(...):
    conv_out = conv(conv_input)        # (B, F, T)
    gated = self_att(conv_out.T)
    unpooled = unpool(gated)
    fusion_sum += unpooled
    conv_input = conv_out              # ← next layer reads THIS layer's conv output
```

All branches use the **same single kernel_size=3**.

### `ufen-mtfn-v2` — Parallel branches with different kernels

All conv branches **independently read from the BiGRU output** and operate in parallel. Each branch has a different kernel size:

```
BiGRU output ──┬──→ Conv(k=1) → ReLU → Self-Att gate → Unpool ─┐
               │                                                  ├→ Element-wise sum → X_fusion
               └──→ Conv(k=5) → ReLU → Self-Att gate → Unpool ─┘
```

```python
# model.py
base = x   # BiGRU output — SHARED input to all branches
for conv, self_att, unpool in zip(self.convs, self.self_atts, self.unpools):
    c = conv(base.transpose(1, 2)).transpose(1, 2)   # ALL branches read `base`
    att_out, _ = self_att(c, c, c, key_padding_mask=mask)
    c = c * att_out
    branch_outputs.append(unpool(c))
fusion = sum(branch_outputs)
```

```python
kernel_sizes = [1, 5]   # k=1 captures pointwise; k=5 captures medium-range context
```

**Architectural correspondence:** The `ufen-mtfn-v2` parallel design more directly matches the paper's wording ("N parallel branches"), while `main`'s sequential design is inspired by the idea that deeper network layers build on earlier extracted features. The parallel design proved more effective.

## 4. Self-Attention Gate in UFEN

### `main` — Custom manual multi-head attention

A bespoke `SelfAttentionGate` class manually implements scaled dot-product attention (QKV projection, reshape, softmax, matmul). No padding mask support.

```python
# src/models/ufen.py — SelfAttentionGate
q = k = v = x.view(B, T, self.num_heads, self.head_dim).transpose(1, 2)
scores = torch.matmul(q, k.transpose(-2, -1)) / self.scale
attn_weights = F.softmax(scores, dim=-1)
attn_out = torch.matmul(attn_weights, v)
return x * attn_out   # gating: X ⊗ Attention(X)
```

### `ufen-mtfn-v2` — `nn.MultiheadAttention` with mask support

Uses PyTorch's built-in `nn.MultiheadAttention` and passes `key_padding_mask` for correct behaviour on padded sequences.

```python
# model.py
self.self_atts = nn.ModuleList([
    nn.MultiheadAttention(conv_dim, n_att_heads, dropout=dropout, batch_first=True)
    for _ in range(self.n_layers)
])
# forward:
att_out, _ = self_att(c, c, c, key_padding_mask=mask)
c = c * att_out   # element-wise gate
```

Both branches implement the same gating mechanism from paper Eq. 5: $X_{self\_att} = X \otimes \text{Attention}(X)$.

## 5. MTFN — Cross-Modal Attention Design

### `main` — Shared per-modality weights (`SharedCrossModalAttention`)

All 6 cross-modal attention pairs **share** the same Q, K, V, and output projections. Each modality has its own projections (W_Q_t, W_Q_a, W_Q_v etc.), but these weights are reused across all pairs that involve that modality.

An extra per-modality **projection layer** (`proj_t`, `proj_a`, `proj_v`) first maps d_m → d_e before QKV computation. `d_e` is a separate hyperparameter (=64) decoupled from `d_m`.

```python
# src/models/mtfn.py — SharedCrossModalAttention
self.proj = nn.ModuleDict({"t": Linear(d_m, d_e), "a": ..., "v": ...})
self.W_Q = nn.ModuleDict({"t": Linear(d_e, d_e), "a": ..., "v": ...})
self.W_K = nn.ModuleDict({"t": Linear(d_e, d_e), "a": ..., "v": ...})
self.W_V = nn.ModuleDict({"t": Linear(d_e, d_e), "a": ..., "v": ...})
self.W_O = nn.Linear(d_e, d_e)          # ONE shared output proj
self.layer_norms = nn.ModuleDict({       # 6 per-pair LayerNorms
    "t_a": ..., "t_v": ..., "a_t": ..., "a_v": ..., "v_t": ..., "v_a": ...
})
```

```
Source: src/models/mtfn.py docstring:
"Key design: Modality projections and Q/K/V weights are SHARED across cross-attention 
pairs, following the paper's Eq. 9 notation where W_i, W_{Q_i}, W_{K_j}, W_{V_j} 
are per-modality (not per-pair)."
```

### `ufen-mtfn-v2` — 6 independent `nn.MultiheadAttention` modules

Each of the 6 directed pairs gets its **own** `nn.MultiheadAttention` instance with fully independent weights. No shared projections. Cross-attention operates directly in `d_m` (no separate d_e). Padding masks are passed via `key_padding_mask`.

```python
# model.py — CrossModalAttention
class CrossModalAttention(nn.Module):
    def __init__(self, d_m, n_heads, att_dropout=0.2):
        self.mha = nn.MultiheadAttention(d_m, n_heads, dropout=att_dropout, batch_first=True)
        self.layer_norm = nn.LayerNorm(d_m)

    def forward(self, x_i, x_j, mask_i=None, mask_j=None):
        out, _ = self.mha(x_i, x_j, x_j, key_padding_mask=mask_j)
        return self.layer_norm(out)

# MTFN instantiates 6 separate CrossModalAttention modules:
make_ca = lambda: CrossModalAttention(d_m, n_cross_heads, att_dropout)
self.ca_t2v = make_ca(); self.ca_t2a = make_ca()
self.ca_v2t = make_ca(); self.ca_v2a = make_ca()
self.ca_a2t = make_ca(); self.ca_a2v = make_ca()
```

**Which interpretation is closer to the paper?** The paper's Eq. 9 uses notation $W_{Q_i}$, $W_{K_j}$, $W_{V_j}$ (per-modality indices), which `main` interprets as shared. However, having fully independent modules per pair (`ufen-mtfn-v2`) gives the model more capacity to specialise each attention direction separately.

## 6. MTFN — Fusion Path to Y_m and Encoder-Decoder Input

Both branches mean-pool each of the 6 cross-attention outputs over time and stack them into a 6-token sequence. However, the intermediate shape differs:

### `main`
```
6 cross-att outputs, mean-pooled → each (B, d_e=64)
→ torch.cat(pooled, dim=-1)        → (B, 6*d_e = 384)
→ fusion_head(fused)               → Y_m  (from flat 384-dim)
→ fused.view(-1, 6, d_e)           → (B, 6, 64)  encoder-decoder input
```

### `ufen-mtfn-v2`
```
6 cross-att outputs, masked_mean   → each (B, d_m=128)
→ torch.stack(..., dim=1)          → (B, 6, 128)  6-token sequence
→ tokens.reshape(B, -1)            → (B, 6*128 = 768)
→ pred_fusion(flat)                → Y_m  (from flat 768-dim)
→ tokens directly fed to encoder   → (B, 6, 128)  encoder-decoder input
```

The `torch.cat → view` vs `torch.stack` approaches are mathematically equivalent, but `ufen-mtfn-v2` avoids the extra reshape by stacking from the start.

## 7. Transformer Encoder-Decoder — FFN Design

### `main` — Dropout inside FFN layers

```python
# src/models/mtfn.py — TransformerEncoderBlock
self.ffn = nn.Sequential(
    nn.Linear(d_model, d_ff),
    nn.ReLU(),
    nn.Dropout(dropout),      # ← dropout between linear layers
    nn.Linear(d_ff, d_model),
    nn.Dropout(dropout),      # ← dropout after second linear
)
```

Additionally, the Encoder and Decoder blocks in `main` use **pre-norm** style (apply LayerNorm *after* the residual add): `x = norm(x + sublayer(x))`.

### `ufen-mtfn-v2` — No dropout inside FFN

```python
# model.py — EncoderBlock
self.ff = nn.Sequential(nn.Linear(d_m, d_ff), nn.ReLU(), nn.Linear(d_ff, d_m))
```

No dropout inside the FFN; dropout is applied to the sublayer output via `self.drop(self.ff(x))` before the residual add.

Both use post-norm style: `x = norm(x + drop(sublayer(x)))`.

## 9. Training Strategy

| Aspect | `main` | `ufen-mtfn-v2` |
|---|---|---|
| Optimizer | Adam | Adam (or AdamW, flag-controlled) |
| LR | 5e-3 (single, all params) | 5e-3 non-BERT + 2e-5 BERT (dual groups) |
| LR scheduler | None (explored ReduceLROnPlateau, CosineAnnealing, SWA in code) | Cosine annealing for both LR groups |
| BERT warmup | N/A (BERT frozen) | 5-epoch linear ramp 0 → 2e-5 |
| Weight decay | 0.0 | 0.0 |
| Grad clipping | None | 1.0 |
| Max epochs | 30 | 50 |
| Early stopping | Patience=8 on **val_loss** | Patience=15 on **dev MAE** |
| SWA | Supported (use_swa flag) | Not implemented |
| Early stop metric | Validation total MSE loss | Dev MAE (primary task metric) |

**Key insight:** `main` uses validation total MSE (sum of 5 sub-task losses) as the early-stopping criterion, which is dominated by the unimodal loss terms. `ufen-mtfn-v2` directly optimises for the metric of interest (MAE on the final $Y'_m$ head), which leads to a better-calibrated best checkpoint.

## 10. Data Preprocessing & Normalization

### `main`
- Fixed T=30 applied during preprocessing; truncates or zero-pads all modalities
- Z-score normalization of audio and visual features using train-set statistics
- Clips extreme COVAREP values (−3.4e38 padding artifacts) before normalization
- Text (BERT) is NOT normalized (already L2-normalized by BERT)

```python
# src/dataset.py
self.audio = np.clip(self.audio, -1e4, 1e4)
self.audio = np.nan_to_num(self.audio, ...)
# then Z-score: (x - mean) / std
```

### `ufen-mtfn-v2`
- Batch-level dynamic padding (each batch padded to longest utterance in that batch)
- No explicit Z-score normalization in `data_loader.py`; relies on the model's normalization (BERT normalizes text; BiGRU adapts to raw scale of audio/visual)
- Generates boolean `mask` tensors alongside data tensors for every modality

## 13. Which Choices Were Correct vs. Debatable

| Decision | `main` | `ufen-mtfn-v2` | Verdict |
|---|---|---|---|
| BERT frozen vs. fine-tuned | Frozen | Fine-tuned | Fine-tuning proved critical (+0.169 MAE). Paper wording is ambiguous but results favour fine-tuning. |
| Conv branches sequential vs. parallel | Sequential (k=3 for all) | Parallel (k=1 and k=5) | Parallel better matches paper's "N parallel branches" wording. Different kernels add multi-scale context. |
| Cross-att shared vs. independent | Shared W_Q/K/V per modality | 6 independent modules | Both are reasonable reads of Eq. 9. Independent gives more parameters; arguably closer to standard MHA practice. |
| Padding masks | Not used | Full mask propagation | Mask propagation is more correct for variable-length data; prevents zero-pad pollution in attention / mean-pool. |
| Early stop metric | val_loss (sum of 5 MSEs) | dev MAE (target metric) | Early-stopping on the evaluation metric directly produces better-calibrated checkpoints. |
| d_e (cross-att dim) | 64 (separate from d_m=64) | 128 (same as d_m) | Both are reasonable; having d_e=d_m avoids an extra projection layer. |
| LR schedule | None | Cosine + warmup | Schedule is strongly beneficial; paper does not specify one but the training dynamics require it. |
| Z-score normalization | Applied to audio/visual | Not applied | `main`'s explicit normalization is more rigorous; `ufen-mtfn-v2` relies on model adaptation. |
| SWA | Supported | Not implemented | SWA could further improve `ufen-mtfn-v2`'s generalization given its dev/test gap. |
